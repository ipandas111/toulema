01

产品背景与目标

每年春秋两季，中国高校有超过 1,000 万应届毕业生同步进入求职市场。校招周期短（春招通常集中在 3–5 月，秋招集中在 9–11 月），岗位信息分散于牛客、Boss直聘、公司官网等多个平台，大量同学依赖飞书/Excel 表格手动记录投递进度，既低效又易遗漏，且无法获得针对具体岗位的个性化简历反馈。

校招通旨在成为求职学生的"作战指挥台"：以可视化看板统一管理多平台投递进度，并通过 AI 能力将简历与目标 JD 进行深度匹配，输出可操作的优化建议，帮助学生用同一份简历的最优呈现形式在短时间内获得更多面试机会。

减少求职过程中的信息焦虑，让学生把精力集中在准备而非记录上

03

用户痛点

  • P1
    进度管理混乱，信息散落多处 投递记录分布在牛客消息、Boss 聊天、邮件、飞书表格，无统一视图，常遗漏跟进
  • P2
    简历与 JD 的匹配度全靠感觉 不知道自己的简历到底差在哪里，收到拒信后无从改进，陷入重复失败的循环
  • P3
    关键时间节点容易遗漏 春秋招期间同时参与多家企业的笔试、面试安排，一旦遗漏即错过，心理压力极大
  • P4
    数据隐私顾虑 不愿将个人求职信息上传第三方平台，担心简历泄露；现有工具多需注册绑定手机号
  • P5
    现有工具学习成本高，校招季用完即弃 Notion/飞书表格配置复杂，学生没耐心设置；希望开箱即用，招完就不用了

采用 Kanban 看板，从左到右代表求职进度递进。

泳道含义典型停留时间优先级
待投递已收藏但尚未提交申请1–7 天P0
已投递简历已提交，等待反馈3–21 天P0
笔试/测评已收到笔试通知1–3 天P0

终面 / 复试高阶面试3–10 天P0

已拒绝主动放弃或收到拒信—P0

6.2 岗位卡片字段

字段类型必填说明
公司名称文本✅显示在卡片标题
岗位名称文本✅支持搜索筛选

优先级标签❌⭐⭐⭐ 三级

6.3 关键交互

  • 拖拽移动：卡片可在泳道间拖动，状态自动更新，支持触屏操作
  • 快速搜索：全局搜索框，实时过滤公司名/岗位名
  • 截止高亮：距截止日 ≤ 3 天的卡片自动显示橙色边框警告
  • 数据导出：一键导出 CSV / JSON，方便迁移或备份
  • 数据导入：粘贴 JSON 或 CSV 即可批量导入历史记录

核心功能 — AI 简历匹配

7.1 功能流程

步骤用户操作系统响应
1粘贴目标 JD 文本（或输入职位关键词）NLP 解析，提取岗位要求关键词、技能、经历要求
2上传简历（PDF/文本粘贴）解析简历结构，提取技能、项目、教育信息
3点击「开始匹配」计算匹配分数，生成分析报告
4查看报告显示匹配度、缺失关键词、建议改写内容
5（可选）AI 生成改写建议针对缺失点生成可直接使用的 bullet point 草稿

7.2 匹配报告结构

匹配总分

0–100 分，分为「关键词匹配」「技能契合」「经历相关度」三个子维度

命中关键词

绿色高亮显示简历中已覆盖的 JD 关键词，让用户直观看到优势

缺失关键词

红色标注 JD 要求但简历未涵盖的词汇，按重要性排序

AI 优化建议

针对缺口给出具体的 bullet point 改写示例，可一键复制

7.3 AI 调用策略

V1.0 方案（MVP）：前端直接调用 Claude API（claude-haiku-4-5 模型，成本低，速度快），API Key 由用户自行输入并仅存于 localStorage，不经过任何服务器。既保护隐私，又无后端运维成本。

V2.0 升级：可选接入自建后端，对 API Key 进行代理，支持免费额度 + 付费订阅模式。

7.4 功能边界（V1.0 不做）

V1.0 不包含：简历在线编辑器、多版本简历对比、面试复盘 AI 辅助、求职数据统计看板。这些列入 V2.0 规划。

08

用户故事

US-01 · 投递记录

作为一名正在春招的应届生，我希望快速添加一条投递记录（公司+岗位+状态），以便在 5 秒内完成录入而不打断我的投递节奏。

验收条件：点击「+ 添加」按钮，填写公司名+岗位名后即可保存，其他字段为选填；保存后卡片立即出现在对应泳道。
US-02 · 进度推进

作为收到面试通知的求职者，我希望把某条投递从「已投递」拖动到「一面」，以便快速更新状态，不需要打开编辑弹窗。

验收条件：拖拽卡片到目标泳道后松手，卡片状态立即更新并持久化至 localStorage。
US-03 · AI 简历分析

作为正在投递 TI 技术销售岗的学生，我希望粘贴 JD 文本和我的简历后，AI 告诉我哪些关键词缺失以及如何改写对应的 bullet，以便在投递前快速调整。

验收条件：分析完成后展示匹配分数（0–100）、命中关键词列表（绿色）、缺失关键词列表（红色）、至少 3 条具体改写建议；分析耗时 ≤ 15 秒。
US-04 · 数据安全

作为对隐私敏感的求职者，我希望无需注册账号、数据全部存在本地，以便放心使用而不担心简历信息泄露。

验收条件：首次打开无需注册；刷新页面数据不丢失（localStorage）；不存在任何向第三方服务器发送投递数据的请求。
US-05 · 数据备份

作为积累了 50 条投递记录的用户，我希望能导出所有数据为 JSON/CSV 格式，以便在换设备时迁移，或在 Excel 中进一步分析。

验收条件：导出后文件包含全部字段；重新导入后数据完整还原，状态与原始一致。

技术架构概要

层级技术选型说明
前端框架原生 HTML / CSS / JS（单文件）零依赖，易于分享和部署；V2 可迁移至 React
数据持久化localStorageV1 无后端；容量 ~10MB，满足 500 条记录

部署Vercel免费托管，自动 HTTPS，CDN 全球加速
版本控制GitHub（公开仓库）开源增加可信度，吸引 Star，利于简历呈现

V1.0 不需要后端的理由

所有数据存本地 → 无需数据库；Claude API Key 用户自填 → 无需代理服务器；静态网页托管 Vercel → 零运维。整体上线成本 < 1 天开发 + ¥30 域名，适合个人 Vibe Coding 快速交付。
